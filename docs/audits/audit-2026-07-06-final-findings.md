### #1 [blocker] **[MONEY]** Manual/direct-buy exposure has NO enforced aggregate ceiling; direct_buy bypasses manual_cap_usd, total_deployed_cap_usd AND max_concurrent (dead helpers, false comment)
- **file:** src/memebot/live/engine.py, dashboard/server/app.py : engine.py:373-417,340-352,384-396; app.py:415-427,370-379
- **effort:** medium
- **what:** direct_buy (the path every human market/limit BUY fires through via OrderBook._fire, and the /api/manual/order route) enforces only a per-order clamp (manual_trade_hard_cap_usd) and an on/off toggle (manual_cap_usd<=0). It never checks aggregate exposure and never calls risk.can_enter() (verified: can_enter's only caller is ingest_call at engine.py:89). Both purpose-built accountants engine.manual_exposure_usd() and app._manual_exposure() are dead (zero callers, grep-verified). So the operator-configured 'total manual exposure ceiling' ($50) does nothing, and direct buys also bypass the algo's total_deployed_cap_usd ($200) and max_concurrent. N direct buys across N mints deploy N x per-order-cap, bounded only by the burner SOL balance — which is the exact hole the moment the burner is funded higher for live. The engine.py:391-392 comment asserting direct buys are 'bounded by the ALGO's own total_deployed_cap/max_concurrent' is false. Additionally direct_buy clamps to manual_trade_hard_cap_usd (dashboard-settable up to $100, app.py:189), 10x the $10 STAKE_HARD_CAP survival ceiling, so a single algo-ridden direct buy can be sized past the researched bust threshold. This holds sequentially (not just as a race): each confirmed direct buy is controller='algo', so even the manual_exposure_usd counter (were it wired) would count $0.
- **fix:** In direct_buy, on the loop thread before submit, call risk.can_enter(prospective_stake=usd) and reject on a non-ok decision (this binds total_deployed_cap/max_concurrent for algo-booked direct buys and is a no-op in paper, preserving equivalence); return the reason as a NON-terminal string so orders retry rather than cancel. Clamp usd to min(manual_trade_hard_cap_usd, STAKE_HARD_CAP_USD) and lower the app.py:189 bound to STAKE_HARD_CAP_USD. If a SEPARATE manual ceiling is truly wanted, FIRST make direct-buy exposure countable (tag origin — they are controller='algo', so the current controller='manual' filter counts $0 and leaks), then enforce exposure+usd<=manual_cap_usd; do NOT wire the current controller='manual' helper. Delete whichever helper stays unused, fix the false engine.py:391-392 comment and the state.py:334 'ceiling' docstring. Add a test: two $10 direct buys FILL (become controller='algo'), cap=$25, assert the 3rd is rejected.

### #2 [blocker] **[MONEY]** total_deployed_cap_usd / max_concurrent are checked only at signal-ingest, never at the capital-committing ENTER — a correlated dip overshoots the configured ceiling with no human action

### #3 [high] RPC endpoint URL (with embedded api-key) leaks into orders.note (dashboard-visible DB) and logs via httpx exception reprs on routine 401/403/429
- **file:** src/memebot/live/jupiter_swap.py, execution.py, engine.py : jupiter_swap.py:109-116; execution.py:166-167; engine.py:267,549-554
- **effort:** small
- **what:** _rpc/quote/build_swap call r.raise_for_status() on SOLANA_RPC_URL. Helius/QuickNode/Alchemy embed the api-key in the URL (path, subdomain, or query) and return 401/403/429 on auth/rate-limit. The httpx.HTTPStatusError str/repr contains the full URL incl. the key (empirically verified). It is caught as repr(e) -> LegResult.error and persisted to orders.note (served to the dashboard) for manual orders and logged for algo. A routine 429 leaks the RPC secret into the SQLite DB + logs, violating the project's 'secrets never logged' invariant. Only fires when live-armed (gated today), so it is a pre-arming secret-hygiene defect, not a fund path.
- **fix:** Convert httpx errors to scrubbed exceptions inside _rpc/quote/build_swap: on HTTPStatusError raise RuntimeError(f'RPC {method} HTTP {e.response.status_code}') from None; on RequestError raise RuntimeError(f'RPC {method} transport error: {type(e).__name__}') from None. Do NOT rely on query-string stripping (Alchemy/QuickNode put the key in the path/subdomain). Add defense-in-depth redaction at the persist/log boundary (replace the os.environ SOLANA_RPC_URL substring and any https?://\S+ token with <rpc> before writing orders.note / record_alert / log).

### #5 [high] A landed SELL whose apply is lost/retried books $0 proceeds (or a landed exit as -100%) — silently corrupts realized P&L on exactly the TP1 leg the strategy's EV lives in; no operator alert
- **file:** src/memebot/live/run.py, executor.py, engine.py : run.py:349-461; executor.py:258-283; engine.py:278-284,300-306
- **effort:** medium
- **what:** Three triggers, one mechanism. (a) Crash/SIGTERM between a confirmed algo SELL and its apply — pipeline.shutdown() drains workers AFTER asyncio.run() closed the loop, so call_soon_threadsafe raises and the confirmed FillResult (with real proceeds) is dropped. (b) A sell that lands but whose confirm times out (~30s) -> rollback -> next candle re-fires. (c) A manual full-close reconciled with no prior legs. _reconcile_submitted_intents resolves ENTER/MANUAL_BUY/MANUAL_SELL but routes algo TP/RIDE_SELL/STOP_OUT/FINALIZE_SUBMITTED to a log-only else. On retry the idempotent executor sizes the already-landed leg to 0 tokens and books proceeds_usd=0; the manual full-close books realized_multiple=0/pnl=-stake. Realized P&L is understated (zeroes the TP1-secure leg config #1's entire positive EV lives in), the daily-loss counter over-accrues (can trip the kill early), and a winning exit can record as a loss — silently, unlike the manual MANUAL_RECON alert. No funds lost (idempotency holds; SOL is in the wallet; error is fail-safe/over-halt direction), but this is the 'book phantom fills' concern and it corrupts the exact measurement metric the mission depends on.
- **fix:** Add an algo-sell branch to _reconcile_submitted_intents mirroring the MANUAL_SELL reconcile: detect the landed leg (real balance < expected pre-leg bag), estimate proceeds = (pre_bag - real) x event.price, route it through the rider-advance path so the state machine transitions correctly and books non-zero proceeds; for a terminal leg close via _finalize_live. NEVER book -stake via record_realized_pnl without a proceeds estimate. Always WARN that exact proceeds need on-chain verification. Root-cause hardening: drain pipeline.shutdown() BEFORE asyncio.run() closes the loop so a confirmed sell's apply is not dropped; and poll _confirm over the full blockhash-validity window with a final getSignatureStatuses(searchTransactionHistory) sweep on timeout.

### #7 [medium] Live algo FINALIZE of an unroutable (rugged) token never confirms -> zombie position retries forever, holds a max_concurrent slot + deployed budget + inflates equity
- **file:** src/memebot/live/engine.py, executor.py : engine.py:158-170,266-269; executor.py:250-264
- **effort:** small
- **what:** In live, finalize_token routes FINALIZE through a real swap that dumps the whole bag. ### [nit] concurrency/ensure-clients-lazy-init-race — _held_tokens calls _ensure_clients() without _state_lock, racing worker lazy-init the lock is supposed to serialize
- file: src/memebot/live/run.py:463-472
- desc: executor._ensure_clients() (executor.py:148-159) lazily assigns self._swap and self._keypair, and the comment at executor.py:127 states _state_lock guards 'lazy client init' so N concurrent workers don't race it. buy/sell_event indeed call _ensure_clients under _state_lock.
